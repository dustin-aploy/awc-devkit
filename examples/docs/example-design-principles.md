# Example Design Principles

These examples are designed to be:

1. **Realistic** — they model bounded roles that a real organization might deploy.
2. **Protocol-aligned** — they use the same declaration shape and pillar vocabulary defined by the AWC protocol.
3. **Role-bounded** — they emphasize what the worker must not do as much as what it may do.
4. **Operationally legible** — they include concrete escalation paths, approval requirements, budget constraints, and measurement criteria.
5. **Demo-friendly** — they pair declarations with task docs and example output artifacts so adopters can see the end-to-end story.

This directory should help teams understand how AWC looks in practice without embedding runtime logic or conformance logic here.
